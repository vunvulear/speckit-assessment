# Implementation Plan: Cloud Latency Monitor

**Branch**: `001-cloud-latency-monitor` | **Date**: 2026-03-26 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/001-cloud-latency-monitor/spec.md`

## Summary

CloudLatency is a Python web application that measures HTTPS latency to all Azure, AWS, and GCP regions worldwide. It consists of three decoupled components: a latency measurement engine, a REST API (FastAPI), and a web UI (vanilla HTML/JS with Chart.js). Data refreshes every 10 seconds and is pushed to the browser via Server-Sent Events.

## Technical Context

**Language/Version**: Python 3.11+
**Primary Dependencies**: FastAPI, uvicorn, aiohttp, sse-starlette, pydantic, Chart.js (frontend)
**Storage**: In-memory (no persistent storage for v1)
**Testing**: pytest, pytest-asyncio, pytest-cov, aioresponses, httpx (TestClient)
**Target Platform**: Any environment (local, VM, container, cloud)
**Project Type**: Web application (backend API + frontend dashboard)
**Performance Goals**: Complete measurement cycle for 100+ regions within 10 seconds; SSE delivery <2s after measurement
**Constraints**: 5s per-request timeout, concurrent async probing, no vendor lock-in
**Scale/Scope**: ~80-150 cloud regions across 3 providers, single-user dashboard (no auth)

**Structure Decision**: Web application layout with `cloudlatency/` as a single Python package containing `engine/`, `api/`, and `ui/` subpackages. This keeps components logically separated while allowing a single deployment unit. The API subpackage is the sole bridge between engine and UI, enabling future replacement of the UI with any REST client.

## Complexity Tracking

No constitution violations to justify. All choices align with simplicity principles:

- No ORM (in-memory data only)
- No SPA framework (vanilla JS)
- No message broker (direct async in-process communication)
- No database (no persistence requirement for v1)
